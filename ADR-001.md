# ADR-001: Backend Architecture for Job Application Platform

## Status
**Approved** | Date: 2026-03-06 | Decision Maker: Caleb Matteis

## Context
Building a job application platform that allows users to apply for jobs and track their application status. Moving from simple contact form to full backend-driven product. Need clean architecture that teaches full-stack fundamentals without over-engineering.

## Problem Statement
- Transform contact form into job application system
- Learn backend fundamentals: HTTP flow, layered architecture, data modeling
- Build maintainable codebase that demonstrates senior engineering principles
- Focus on product value over architectural complexity

## Decision
**Layered Architecture with Repository Pattern**

### Core Technology Stack
- **Frontend:** React 19.2.0 + TypeScript + Vite
- **Backend:** Express.js + TypeScript
- **Database:** Prisma ORM + SQLite (local dev)
- **Architecture:** Routes → Controllers → Services → Repositories

### Architecture Principles
1. **Separation of Concerns:** Each layer has single responsibility
2. **Simple Repository Pattern:** Clean data access without magical injection
3. **Audit Trail:** Status history table for workflow tracking
4. **Business Logic in Services:** Controllers stay thin
5. **Type Safety:** End-to-end TypeScript for reliability

## What We're Building
**Core Domain Models:**
- User (applicants, recruiters, admins)
- Job (job postings with details)
- Application (user applies to job)
- ApplicationStatusEvent (audit trail for status changes)

**Key Workflows:**
- Job creation and management
- Application submission process
- Status tracking with audit trail
- Role-based access control

## What We're NOT Doing Yet
- **Multi-tenancy** (single-tenant for now - can add later)
- **Event sourcing** (simple status history sufficient)
- **Row-level security** (app-level authorization adequate)
- **Enterprise patterns** (outbox, dead letter queues, CQRS)
- **Microservices** (monolith first, extract later if needed)

**Rationale:** Focus on mastering fundamentals before adding enterprise complexity. Each excluded pattern can be added in future phases when business need justifies the complexity.

## Success Metrics
- **Functionality:** Complete job application workflow works end-to-end
- **Code Quality:** Clear request/response flow with proper error handling
- **Architecture:** Clean separation of concerns across all layers
- **Learning:** Deep understanding of backend fundamentals gained
- **Maintainability:** New features can be added without major refactoring

## Consequences
### Positive
- Faster development and learning cycle
- Easier to debug and test
- Clear upgrade path to enterprise patterns
- Demonstrates product-focused engineering mindset

### Risks & Mitigation
- **Scalability concerns** → Monitor and refactor when needed, architecture supports evolution
- **Missing audit requirements** → Status events provide audit trail foundation
- **Security gaps** → App-level auth sufficient for current needs, can add RLS later

## Future Evolution Path
**Phase 2:** Enhanced auth, file uploads, advanced workflows
**Phase 3:** Performance optimization, caching, monitoring
**Phase 4:** Multi-tenancy and enterprise patterns (if business requires)
**Phase 5:** Microservices extraction (if scale demands)

## Technical Implementation

### Database Schema
```prisma
// ==========================================
// CORE DOMAIN MODELS
// ==========================================

enum UserRole {
  APPLICANT    // Job seekers
  RECRUITER    // Company recruiters
  ADMIN        // Platform administrators
}

enum ApplicationStatus {
  SUBMITTED
  UNDER_REVIEW
  INTERVIEW_SCHEDULED
  INTERVIEWED
  OFFER_MADE
  ACCEPTED
  REJECTED
  WITHDRAWN
}

// ==========================================
// 1. USERS & AUTHENTICATION
// ==========================================

model User {
  id           String        @id @default(cuid())
  email        String        @unique
  name         String?
  role         UserRole      @default(APPLICANT)
  isActive     Boolean       @default(true)
  lastLoginAt  DateTime?

  // Relationships
  applications Application[]
  jobs         Job[]         // Jobs posted by recruiters
  statusEvents ApplicationStatusEvent[]

  // Audit
  createdAt    DateTime      @default(now())
  updatedAt    DateTime      @updatedAt

  @@index([email])
  @@index([role])
}

// ==========================================
// 2. JOB MANAGEMENT
// ==========================================

model Job {
  id           String        @id @default(cuid())
  title        String
  company      String
  description  String
  location     String?
  salary       String?
  isActive     Boolean       @default(true)
  postedById   String

  // Relationships
  postedBy     User          @relation(fields: [postedById], references: [id])
  applications Application[]

  // Audit
  createdAt    DateTime      @default(now())
  updatedAt    DateTime      @updatedAt

  @@index([isActive, createdAt])
  @@index([company])
}

// ==========================================
// 3. APPLICATION WORKFLOW
// ==========================================

model Application {
  id           String                    @id @default(cuid())
  userId       String
  jobId        String
  status       ApplicationStatus         @default(SUBMITTED)
  coverLetter  String?
  resumeUrl    String?
  notes        String?

  // Relationships
  user         User                      @relation(fields: [userId], references: [id])
  job          Job                       @relation(fields: [jobId], references: [id])
  statusHistory ApplicationStatusEvent[]

  // Audit
  createdAt    DateTime                  @default(now())
  updatedAt    DateTime                  @updatedAt

  // Business Rules
  @@unique([userId, jobId]) // Can't apply twice to same job
  @@index([userId, status])
  @@index([jobId, status])
}

// ==========================================
// 4. AUDIT TRAIL & WORKFLOW HISTORY
// ==========================================

model ApplicationStatusEvent {
  id            String            @id @default(cuid())
  applicationId String
  fromStatus    ApplicationStatus?
  toStatus      ApplicationStatus
  notes         String?
  changedById   String?

  // Relationships
  application   Application       @relation(fields: [applicationId], references: [id])
  changedBy     User?             @relation(fields: [changedById], references: [id])

  // Audit
  createdAt     DateTime          @default(now())

  @@index([applicationId, createdAt])
}
```

### API Architecture
```
Routes (HTTP layer)
├── /api/auth/*          → AuthController
├── /api/jobs/*          → JobController
├── /api/applications/*  → ApplicationController
└── /api/users/*         → UserController

Controllers (Request/Response handling)
├── Validate request data
├── Extract user context
├── Call appropriate service
└── Format response

Services (Business logic)
├── JobService
├── ApplicationService
├── UserService
└── AuthService

Repositories (Data access)
├── JobRepository
├── ApplicationRepository
├── UserRepository
└── Base repository patterns
```
