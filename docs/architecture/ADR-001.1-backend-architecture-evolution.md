# ADR-001.1: Backend Architecture Evolution - DDD Implementation

## Status
**Amendment to ADR-001** - March 22, 2026

## Context
During implementation (Weeks 1-2), the original backend architecture decision (ADR-001) evolved significantly. The actual implementation demonstrates more sophisticated Domain-Driven Design patterns than originally specified. This amendment documents the architectural reality and rationale for the evolution.

## What Changed from ADR-001

### 1. **Repository Pattern Evolution**

**Original ADR-001 Plan:**
```typescript
// Simple repository without abstraction
class JobRepository {
  constructor(private prisma: PrismaClient) {}

  async findById(id: string): Promise<Job | null> {
    return this.prisma.job.findUnique({ where: { id } });
  }
}
```

**Actual Implementation:**
```typescript
// Advanced generic repository with type safety
export abstract class AbstractBaseRepository<
  T,
  CreateInput,
  UpdateInput,
  UniqueInput extends { id?: string },
  WhereInput,
  FindUniqueArgs extends { where: UniqueInput }
> {
  constructor(
    protected modelDelegate: BaseDelegate<T, CreateInput, UpdateInput, UniqueInput, WhereInput, FindUniqueArgs>
  ) {}

  async findById(id: string): Promise<T | null> {
    return this.modelDelegate.findUnique({ where: { id } as UniqueInput } as FindUniqueArgs);
  }
  // ... comprehensive CRUD operations with type safety
}

// Domain interface
export interface JobRepository {
  findById(id: string): Promise<Job | null>;
  findActiveJobs(): Promise<Job[]>; // Business-specific method
  create(data: Prisma.JobCreateInput): Promise<Job>;
}

// Infrastructure implementation
export class PrismaJobRepository
  extends AbstractBaseRepository<Job, Prisma.JobCreateInput, ...>
  implements JobRepository
{
  async findActiveJobs(): Promise<Job[]> {
    return this.modelDelegate.findMany({ where: { isActive: true } });
  }
}
```

**Why This Evolution Happened:**
- **Type Safety:** Generic abstractions eliminate runtime errors
- **Code Reuse:** Abstract base eliminates 80% of boilerplate across entities
- **Domain Separation:** Clear boundary between business logic and infrastructure
- **Testability:** Interface-based design enables easy mocking

### 2. **Database Technology Decision Update**

**ADR-001 Original:**
```prisma
datasource db {
  provider = "sqlite" // For local development
}
```

**Actual Implementation:**
```prisma
datasource db {
  provider = "postgresql" // From day 1
}
```

**Rationale for Change:**
- **Production Parity:** Avoid SQLite → PostgreSQL migration issues
- **Advanced Features:** Need for robust indexing and JSON support
- **Docker Development:** PostgreSQL container provides production-like environment
- **Learning Value:** PostgreSQL skills more valuable than SQLite experience

### 3. **Enhanced Domain Modeling**

**Original Schema (ADR-001):**
```prisma
enum ApplicationStatus {
  SUBMITTED
  UNDER_REVIEW
  INTERVIEW_SCHEDULED
  INTERVIEWED
  OFFER_MADE
  ACCEPTED
  REJECTED
  WITHDRAWN
}

model User {
  id    String @id @default(cuid())
  email String @unique
  name  String?
  role  UserRole @default(APPLICANT)
  // Basic fields only
}
```

**Evolved Schema (Current Implementation):**
```prisma
enum ApplicationStatus {
  SUBMITTED      // Initial application
  RECEIVED       // Acknowledged by recruiter
  SCREENING      // Initial review phase
  INTERVIEWING   // Active interview process
  OFFER_EXTENDED // Offer made
  HIRED          // Successful completion
  REJECTED       // Not moving forward
}

enum OnboardingStatus {
  NOT_STARTED
  IN_PROGRESS
  COMPLETED
}

model User {
  id               String           @id @default(cuid())
  email            String           @unique
  name             String?
  role             UserRole         @default(CANDIDATE) // Renamed from APPLICANT
  onboardingStatus OnboardingStatus @default(NOT_STARTED) // New field

  // Enhanced relationships
  applications     Application[]
  postedJobs       Job[]            @relation("JOBPOSTER")
  statusChanges    ApplicationStatusEvent[]

  createdAt        DateTime         @default(now())
  updatedAt        DateTime         @updatedAt
}

model Job {
  // Enhanced indexing strategy
  @@index([postedById])
  @@index([company])     // Business requirement: search by company
  @@index([location])    // Business requirement: location filtering
}

model Application {
  // Business rule enforcement
  @@unique([userId, jobId]) // Prevent duplicate applications

  // Performance-optimized indexes
  @@index([status])         // Recruiter dashboard queries
  @@index([userId])         // Candidate dashboard queries
  @@index([jobId])          // Job-specific application lists
}
```

**Domain Evolution Rationale:**
- **Business Alignment:** Status names match actual recruiting workflows
- **Performance Optimization:** Strategic indexing based on query patterns
- **Data Integrity:** Unique constraints prevent business rule violations
- **Future-Proofing:** Onboarding status supports user engagement tracking

### 4. **Service Layer Implementation**

**Architecture Reality:**
```typescript
// Clean dependency injection through constructor
export class JobService {
  constructor(private jobRepo: JobRepository) {} // Interface, not concrete class

  async getAllOpenRoles(): Promise<Job[]> {
    return this.jobRepo.findActiveJobs(); // Business method on repository
  }

  async postNewJob(data: CreateJobDTO): Promise<Job> {
    // Business Rule: New jobs start as active
    return this.jobRepo.create({
      ...data,
      isActive: true, // Enforced at service layer
      postedBy: { connect: { id: data.postedById } }
    });
  }
}
```

**Key Improvements Over ADR-001:**
- **Interface Dependency:** Services depend on interfaces, not implementations
- **Business Rule Enforcement:** Logic centralized in service layer
- **Type Safety:** Shared DTOs ensure consistency across layers
- **Clean Architecture:** Clear separation of concerns maintained

## Lessons Learned

### 1. **Architecture Emerges Through Implementation**
The sophisticated patterns weren't over-engineered upfront - they emerged as solutions to real problems encountered during development.

### 2. **TypeScript Generics Enable Elegant Abstractions**
The generic repository pattern wouldn't be possible without TypeScript's advanced type system.

### 3. **Domain-Driven Design Scales Down**
DDD principles are valuable even in smaller applications and improve long-term maintainability.

### 4. **Infrastructure Abstraction Pays Dividends**
The repository interfaces make the codebase database-agnostic and highly testable.
